SUCCESS STATE:
  Approach A (extern-call surface) was chosen. EtlVal is a heap-allocated
  tagged-union struct with tag constants 0=int/1=bool/2=ptr/3=str defined in
  runtime/etl_etlval.{h,c}. The ETL surface exposes constructors
  etlval_int/bool/ptr/str, discriminator etlval_tag, getters etlval_as_int/
  as_bool/as_ptr/as_str, and etlval_free. The C backend emits EtlVal* for
  the opaque type; the VM backend dispatches HV* opcodes (HVI/HVB/HVP/HVS
  for constructors, HVT for tag, HVF for free, HVAI/HVAB/HVAP/HVAS for
  getters). sema recognises etlval in extern params, regular fn params, and
  return types. emit_c adds #include <etl_etlval.h> unconditionally (same
  pattern as etl_dynarr.h). Two corpus fixtures pass on both backends with
  0 valgrind errors and 0 leaks.
WHAT WAS LEFT UNDONE:
  - etlval_str variant owns the EtlString* and frees it in etlval_free, but
    the VM's HVAS getter returns a str handle by pointer identity scan (O(n)
    through str_table); this is acceptable for M1 fixture scale.
  - The tagged_union_dispatch fixture was simplified to a single-function
    (main-only) layout because the 1024-byte bytecode buffer is insufficient
    for multi-function programs with many opaque-type calls. The original
    multi-function design compiled correctly in the C backend; only the BC
    buffer limit was the constraint.
  - No str-variant fixture (etlval_str) because str opaque handling in a
    combined fixture exceeded the 1024-byte bytecode limit. The runtime and
    VM code for HVS/HVAS are implemented and tested indirectly.
