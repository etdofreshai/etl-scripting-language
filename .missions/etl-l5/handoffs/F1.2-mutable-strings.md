SUCCESS STATE:
  The str opaque type (TY_STR=5) is now recognized by sema, accepted in extern
  param and return positions, and emitted by the C backend as EtlString* (from
  runtime/etl_string.h). The runtime provides str_new (copies from void* literal
  into heap), str_len, str_concat, str_at, str_eq, str_free in
  runtime/etl_string.c. The VM backend handles HS* sub-opcodes (HSN, HSL, HSC,
  HSA, HSE, HSF) inside the existing H opcode block, backed by a str_table of
  EtlString* handles (max 64 slots, auto-freed on exit). Three corpus fixtures
  (tests/c1_corpus/string_heap_{basic,concat,eq}.etl) exercise the full surface
  and pass with exit 0 on both the C backend (scripts/c1_string_equiv_smoke.sh)
  and the VM backend (scripts/c1_vm_string_smoke.sh). Valgrind reports 0 errors
  and 0 leaks on all string fixtures. The emit_c preamble now always emits
  #include <etl_string.h> alongside <stdbool.h>; all affected smoke scripts were
  updated to pass -I runtime and/or link etl_string.c. Makefile ETL_RUNTIME now
  includes runtime/etl_string.c.
WHAT WAS LEFT UNDONE:
  - compiler0 (Python) does not support the str type; string fixtures cannot be
    compiled by compiler0 and are therefore excluded from c1_equiv_smoke.sh
    (which requires a compiler0 baseline). A dedicated c1_string_equiv_smoke.sh
    uses compiler-1 C backend vs VM backend instead.
  - The str type is not supported in regular fn parameter lists (only extern fns);
    sema_count_i32_params does not accept str.
  - The VM str_new always creates an empty string (ignores the ptr content)
    because the VM has no way to dereference arbitrary pointers to read string
    literals. Programs relying on meaningful string content via the VM will not
    work unless they build content through str_concat or other operations.
  - #include <etl_string.h> is unconditionally emitted even for programs that do
    not use the str type; a future optimization could emit it conditionally.
DISCOVERED ISSUES:
  - emit_bytecode AN_EXPR_STMT only intercepted etl_free; str_free as a statement
    fell through to C<str_free>; which VM could not resolve (-34). Fixed by adding
    str_free interception in the EXPR_STMT handler.
  - The H opcode restructuring: the original HA;/HF; format uses one sub-char
    then ';'; HS* uses one 'S' then a second sub-char then ';'. The two-char
    lookahead required merging the HS* handling into the existing H block rather
    than a separate else-if branch.
  - str_at return type conflict: ETL i8 return maps to int in the emitted extern
    prototype, but etl_string.h originally declared int8_t. Resolved by changing
    etl_string.h to return int.
  - str_new parameter: ETL ptr maps to void* in emitted C; etl_string.h had
    const void*. Resolved by removing const.
  - The emit_c preamble include #include "etl_string.h" with quoted path caused
    ETL string literal truncation (ETL does not support escaped quotes inside
    string literals). Resolved by using angle-bracket form #include <etl_string.h>.
