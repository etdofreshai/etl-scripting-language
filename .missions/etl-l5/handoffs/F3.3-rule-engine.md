SUCCESS STATE:
  F3.3-rule-engine ships a working runtime rule-engine demo. The host program
  (examples/cli/config_rules.etl, AOT-compiled via C backend) reads a rule
  script file (examples/cli/config_rules.rules.etl) at startup and processes
  5 input records from examples/cli/config_rules.input. For each record the
  host calls etl_build_rule_source() to generate a specialized ETL source
  string (wrapping the rule's evaluate() function in a main() that passes the
  record value as a literal), compiles it via etl_compile_module, and runs it
  via etl_run_main_i32 with ETL_VM_ETL set, so the rule bytecode runs through
  bin/etl-vm-etl. The rule returns a decision code (0=ALLOW, 1=DENY,
  2=REVIEW) as a process exit code; the host prints "record N: decision D".
  The smoke script (scripts/cli_config_rules_smoke.sh) asserts output matches
  examples/cli/config_rules.expected and verifies the [etl_host] VM log
  appears on stderr. New runtime helpers: etl_parse_csv_record,
  etl_build_rule_source, etl_write_record, etl_read_record in
  runtime/etl_runtime.c.
WHAT WAS LEFT UNDONE:
  - The rule file (config_rules.rules.etl) has no // comments because the c1
    bytecode driver does not support // comments (while c0 does). If c1 gains
    comment support in a future feature, the rule file can be documented.
  - bin/etl check on config_rules.etl is skipped in examples_cli_smoke.sh
    because it uses c0-style // comments. A c1-compatible host format would
    be a future improvement.
  - etl_write_record and etl_read_record were added to etl_runtime.c but are
    not used by the final implementation (the simpler etl_build_rule_source
    approach was chosen instead). They can be removed in a cleanup pass.
DISCOVERED ISSUES:
  - The documented "once-per-process limit" on etl_compile_module in
    etl_host.h/etl_host.c is stale. The implementation uses temp files
    (not /dev/stdin) for both source and bytecode, and works correctly for
    any number of calls within a process. Verified empirically.
  - The ETL VM subprocess (bin/etl-vm-etl) cannot call C extern functions
    from bytecode programs. The C bytecode opcode dispatches only to ETL
    functions in the compiled function table, not to linked C symbols.
    Therefore rule scripts running in the VM cannot do file I/O or printing;
    they can only compute and return a result via process exit code.
  - The c1 lexer does not handle // comments, causing "lex error" if
    config_rules.etl (which uses // for documentation) is passed to
    bin/etl check. The rule file is written comment-free to avoid this.
