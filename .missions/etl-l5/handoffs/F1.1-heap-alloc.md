SUCCESS STATE:
  The ptr opaque type is now recognized by sema (TY_PTR=4), emitted as void*
  by the C backend (for extern params, extern return, and local let bindings),
  and handled in the VM via new HA;/HF; opcodes backed by calloc/free with a
  64-slot bounded allocation table that auto-frees on clean exit. The fixture
  tests/c1_corpus/heap_alloc_basic.etl (extern fn etl_alloc/etl_free, one
  alloc+free of 4 bytes, returns 0) passes under both the C backend
  (scripts/c1_equiv_smoke.sh, 34/34) and the VM backend
  (scripts/c1_vm_heap_alloc_smoke.sh). Valgrind reports 1 alloc, 1 free,
  0 errors, 0 leaks on the C-backend binary. AN_EXPR_STMT support was added
  to emit_bytecode.etl as a prerequisite for emitting void extern calls.
WHAT WAS LEFT UNDONE:
  - ptr arithmetic and dereference are intentionally excluded (non-goal per spec)
  - ptr as a regular fn parameter type (sema_count_i32_params does not accept ptr;
    only extern fns support ptr params in this implementation)
  - etl_is_null is not yet available in the bytecode/VM backend (no BN; opcode)
  - HA; opcode does not yet handle size<=0 specially (pushes 0 handle, which is
    consistent with the null convention)
  - The etl_vm.h description still says "does not allocate" — that sentence is now
    stale for programs using HA;/HF; but was left as-is to not break existing
    documentation contracts
DISCOVERED ISSUES:
  - ETL strings in Python-generated harness code require literal \n (two chars),
    not actual newline — Python string escape leaked into an ETL source file and
    caused a parse error; fixed by using raw Python strings
  - emit_bytecode_emit_stmt did not handle AN_EXPR_STMT before this feature;
    void extern calls (like etl_free used as a statement) would have returned
    EMIT_ERR_UNSUPPORTED for any program that called void externs as statements
