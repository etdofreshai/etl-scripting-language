SUCCESS STATE:
  examples/cli/calculator.etl is now a stdin REPL that reads arithmetic
  expressions line by line, evaluates them, and prints results to stdout.
  It handles +, -, *, /, parentheses, and integer constants. Malformed
  input prints an error to stderr and continues without crashing. EOF
  causes exit 0. Implementation uses two new C runtime helpers:
  etl_read_line (reads one line, returns byte count or -1 on EOF) and
  etl_calc_line/etl_calc_eval (recursive-descent parser for arithmetic
  expressions, prints result internally, prints error to stderr on
  malformed input). The ETL file compiles via the C backend (build_etl.sh)
  to a native binary. scripts/cli_calculator_smoke.sh feeds
  examples/cli/calculator.input (covering simple arithmetic, parentheses,
  nested expressions, and a malformed line) and diffs stdout against
  examples/cli/calculator.expected. The smoke is wired into make
  examples-cli. scripts/examples_cli_smoke.sh was updated to pass EOF
  to the new REPL and expect exit 0 instead of the old hardcoded exit 9.
WHAT WAS LEFT UNDONE:
  - Float support (out of scope per constraints: integer-only is fine).
  - Unary minus on named variables (only on literals and sub-expressions
    currently); not required by acceptance.
  - The ETL VM path (bin/etl run) for the calculator is not independently
    tested with expressions -- that is F3.3 territory.
DISCOVERED ISSUES:
  - ETL expression statements may only call void extern functions; this
    required adding a void wrapper etl_calc_line around the int32_t-
    returning etl_calc_eval. This is a known ETL language constraint.
  - The old examples_cli_smoke.sh expected calculator to exit 9 (the
    hardcoded expression result). That test was updated to feed EOF and
    expect exit 0, matching the new REPL semantics.
