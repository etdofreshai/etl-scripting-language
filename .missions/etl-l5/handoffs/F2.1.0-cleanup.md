## What Was Done

### Finding 1 — Harden host_runtime_compile_smoke.sh (falsifiability)

**Problem:** The smoke test only checked exit 0 from the host program.  It could not detect if the ETL VM code path was silently bypassed (e.g. ETL_VM_ETL routing fell back to the C oracle VM without error).

**Fix (commit 2ffdb0e):**

1. `runtime/etl_host.c` — Added a single `fprintf(stderr, ...)` on the ETL VM code path immediately after the `ETL_VM_ETL` env check succeeds:
   ```c
   fprintf(stderr, "[etl_host] using ETL VM at %s\n", etl_vm);
   ```
   The log is emitted only on the ETL VM branch; the C oracle VM fallback path is silent.  Production output is unaffected unless `ETL_VM_ETL` is set.

2. `scripts/host_runtime_compile_smoke.sh` — Redirected stderr to a temp file during the host program run.  After checking exit 0, greps the log for `[etl_host] using ETL VM at`.  Fails with a descriptive message if the line is absent, proving ETL VM routing actually executed.

### Finding 2 — Document etl_compile_module sequential limitation

**Problem:** There was no documented warning that `etl_compile_module` cannot be called more than once per process.  The subprocess bytecode driver reads `/dev/stdin`, which is consumed on first use.

**Fix (commit 39e77d9):**

- `runtime/etl_host.h` — Added LIMITATION comment block above the `etl_compile_module` declaration.
- `runtime/etl_host.c` — Mirrored the same LIMITATION comment block at the implementation site.

### Finding 3 — F2.1 handoff recovery

**Problem:** The F2.1 handoff document existed only at `/home/node/workspace/.missions/etl-l5/handoffs/F2.1-vm-core-in-etl.md` (outside the repo), not tracked in version control.

**Fix (commit 4ea0392):** Copied the handoff into `.missions/etl-l5/handoffs/F2.1-vm-core-in-etl.md` inside the repo.  The document references commit 50d8c47 (F2.1 delivery) and documents: compiler1/vm.etl (integer arithmetic, locals, branches, calls, return, HA/HF), opaque-type stack-discipline stubs (HS*/HD*/HV*) deferred to F2.1.1.

## Architecture Notes

- The stderr log in `etl_host.c` is guarded by the ETL VM code path — no output is produced when the C oracle VM is used.  Equivalence gates (`vm-equivalence`, `triple_equiv_smoke`) do not set `ETL_VM_ETL` and are therefore unaffected.
- The LIMITATION comment on `etl_compile_module` is documentation-only.  The underlying one-call-per-process constraint is not fixed here; it is tracked for a future feature (likely requires replacing the subprocess stdin piping with a file-based or socket-based IPC protocol).
