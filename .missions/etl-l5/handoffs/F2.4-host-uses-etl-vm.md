SUCCESS STATE:
  etl_run_main_i32 in runtime/etl_host.c now routes through bin/etl-vm-etl
  (Approach A: subprocess) when the ETL_VM_ETL environment variable is set.
  Bytecode is written to a mkstemp temp file, etl-vm-etl is invoked with stdin
  redirected to that file, and the process exit code is captured as the i32
  result. The C oracle VM (etl_vm_run_main_i32) remains as a fallback when
  ETL_VM_ETL is unset, preserving vm-equivalence and triple-equivalence gates.
  Two new runtime support files were added: runtime/etl_host_etl_api.c provides
  ETL-callable C wrappers (host_compile_module, host_run_main_i32) with array
  pointer signatures compatible with ETL extern-fn declarations.
  tests/host/runtime_compile_run.etl is an ETL program that calls these wrappers
  end-to-end, compiling "fn main() i32 ret 6 * 7 end" and asserting result==42.
  scripts/host_runtime_compile_smoke.sh builds all components and runs the test.
WHAT WAS LEFT UNDONE:
  Approach B (library link: compile vm.etl into a static object linked in-process)
  is deferred to M3. The subprocess boundary adds fork+exec cost per run call;
  acceptable for M2 but will need re-evaluation for sample apps with frequent
  host-bridge calls. The sequential etl_compile_module limitation (second call in
  the same process fails because the bytecode driver reads /dev/stdin which is
  shared with the parent process's stdin) is a known pre-existing issue unrelated
  to F2.4; the ETL test and smoke script use single-call-per-process invocation
  to avoid it.
DISCOVERED ISSUES:
  etl_compile_module cannot be called twice within the same process: the second
  system() invocation of the bytecode driver reads /dev/stdin but the file
  descriptor state is already exhausted from the parent. This is a pre-existing
  limitation in the original etl_host.c design. The smoke script and ETL test
  work around this by doing one compile+run per process invocation, matching the
  pattern of the existing c1_runtime_compile_smoke.sh. Tracked for future fix.
