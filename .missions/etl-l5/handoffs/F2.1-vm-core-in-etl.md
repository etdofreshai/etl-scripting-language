## What Was Done

### Deliverables

1. **`compiler1/vm.etl`** — Full bytecode interpreter in ETL (1718 lines).
   - Instruction dispatch loop over the ETLB1 bytecode format.
   - Integer arithmetic: `+;` `-;` `*;` `/;` `%;`
   - Comparisons: `q;` `n;` `<;` `l;` `>;` `g;` `!;`
   - Locals (L/store): `L<n>;` load, `L<n>=;` store, per-frame 32-slot array.
   - Labels and branches: `:<n>;` (no-op), `J<n>;` (jump), `F<n>;` (jump-if-false).
   - Function table: `T<n>;` `D<name>,<argc>;` — parsed into parallel arrays.
   - Function calls: `C<name>;` with frame push and arg population.
   - Return: `R;` — callee return to stack, or top-level final return.
   - Body markers: `@<name>;` — no-op at runtime.
   - Heap handle tracking: `HA;` `HF;` (alloc/free lifecycle tracked as handle counters).
   - String, dynarr, etlval op stubs: `HS*` `HD*` `HV*` — stack discipline maintained; handles tracked as counters; actual string/array operations stubbed (see WHAT WAS LEFT UNDONE).
   - Reads bytecode from stdin via `etl_read_file("/dev/stdin", ...)`.
   - Exits with `etl_exit(result)` where result is the i32 return of `main`.

## Architecture Notes

- The ETL VM uses **only compiler0/c1 language surface** — no new ETL syntax.
- The C VM (`runtime/etl_vm.c` + `runtime/etl_vm.h`) is **completely untouched**.
- Instruction pointer `ip` is stored as a 1-element `i32[1]` array (ETL's out-parameter idiom for mutable scalars).
- Function table stored as parallel flat arrays: `f_name i8[1024]` (32 funcs × 32 chars), `f_nlen i32[32]`, `f_argc i32[32]`, `f_ip i32[32]`.
- Call frames stored as `frames_rip i32[32]` and `frames_locals i32[1024]` (32 frames × 32 locals).
- ETL while-loop break idiom: `running2 = 0` flag instead of `break`.
- `vm_find_body` and `vm_find_label` do linear scans (no index pre-building) — safe for the test corpus; for large programs `vm_find_label` is O(n) per branch, totaling O(steps×n).

## What Was Left Undone

**HS*/HD*/HV* ops are stub-only.** The ETL VM tracks handle counters for strings, dynarrs, and etlvals, and maintains correct stack discipline (correct pops/pushes), but does not actually call the underlying `str_*`, `dynarr_*`, `etlval_*` runtime functions. This means programs that depend on real string content, dynarr element values, or etlval tags will produce incorrect (zeroed) results.

**Reason:** The ETL language surface has no mechanism to call arbitrary C functions beyond explicitly declared `extern fn` declarations. The current vm.etl would need explicit extern declarations for every `str_*`/`dynarr_*`/`etlval_*` function, and those functions use C pointer types not representable in ETL's type system without a bridge layer. This is the scope-cap described in the feature spec (deferred as F2.1.1 or F2.3).

**HA/HF** are also stub-tracked (handle counter only). The actual `calloc`/`free` are not called from vm.etl. This means programs that use the allocated memory indirectly through other ops will not work correctly.

**Provider note:** Implemented entirely without provider fallback. No rate limits hit.
